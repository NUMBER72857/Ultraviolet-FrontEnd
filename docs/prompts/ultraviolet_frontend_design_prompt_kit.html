Frontend Design Prompt Kit

AstroPay-inspired prompts for Ultraviolet's student booking site, Stellar USDC checkout, auth and identity flows, and admin operations dashboard.

Reference AstroPay's current navigation, global payment trust signals, exchange widgets, and business payment pages.
Repo Structure ultraviolet-backend, ultraviolet-frontend, ultraviolet-ops.

Prepared on May 17, 2026. AstroPay reference pages reviewed: homepage, business page, accept-payments page, and developer docs landing page.

Read This First

Ruthless boundary: AstroPay and the earlier Stargate idea are references, not things to clone. Ultraviolet is an education operations product with backend-verified Stellar payments. Do not turn it into a token, NFT, credential, or Soroban demo before payment creation, verification, reconciliation, auth, and admin auditability work.

AstroPay is useful because it shows how a money product makes global payments feel simple: clear navigation, trust proof, exchange-rate interaction, concise feature strips, and business payment positioning. Ultraviolet should borrow the clarity, not the brand. The UI must serve students booking education consulting services and admins reconciling payment status against backend-owned truth.

Repo Structure Reference

Use the three-repo reference structure as a separation-of-concerns model, but keep Ultraviolet's real split. A contract repo is not justified for the MVP.

ultraviolet-backend

Rust backend for payment truth, auth, identity, reconciliation, admin APIs, and audit logs.

  • Owns invoice/payment state.
  • Verifies Stellar facts server-side.
  • Protects student PII.

ultraviolet-frontend

Next.js frontend for public booking, checkout, admin dashboard, and auth flows.

  • Displays backend status.
  • Collects student intake.
  • Never decides payment truth.

ultraviolet-ops

Deployment, environment contracts, runbooks, production gates, and incident response.

  • Database and service orchestration.
  • Reconciliation runbooks.
  • Security and launch checklists.
Header rule: brand docs and README headers as NUMBER / Ultraviolet until final brand identity is locked.

AstroPay Reference Audit

Patterns Worth Reusing

  • Top-level navigation that separates user types and product lines.
  • Dark hero with a direct borderless-money promise.
  • Exchange-rate widget near the hero as proof of utility.
  • Trust proof near conversion CTAs.
  • Simple action verbs for product sections.
  • Business page focused on reach, cost, speed, and developer integration.

Ultraviolet Must Improve

  • AstroPay is broad fintech. Ultraviolet needs education-intake specificity.
  • Ratings are not enough. Show backend status, payment status, and admin auditability.
  • Checkout must explain Stellar verification without exposing students to protocol noise.
  • Auth and identity need first-class UI, not a login modal afterthought.
  • AI output is advisory metadata only, never payment or eligibility truth.
  • No frontend state may downgrade or verify payment status.

Prompt 00 - Preflight Boundary

00-A / Audit Before Building UI
Audit the Ultraviolet frontend task before writing code.

Repos:
- ultraviolet-backend owns auth, payment truth, reconciliation, applicant records, and audit logs.
- ultraviolet-frontend owns public booking UI, checkout UI, admin UI, and display-only status.
- ultraviolet-ops owns environment contracts, deployment, and runbooks.

Required output:
- Which repo owns the change.
- Which backend endpoint or mock contract the frontend depends on.
- Which PII fields are touched.
- Which payment states are display-only.
- Which auth role can access the screen.
- Which AstroPay pattern is being adapted.
- Which pattern must be rejected because it is brand-specific, consumer-only, or unsafe.

Hard rules:
- Do not use AstroPay logos, images, exact text, or protected trade dress.
- Do not create a contract repo for the MVP.
- Do not mark payments paid, verified, settled, or failed from frontend-only state.
- Do not expose applicant/admin records through public routes.

01-B / Typography, Layout, Motion
Build Ultraviolet's typography, spacing, radius, and motion primitives.

Requirements:

Output:

- CSS utility layer.
- Storybook page showing typography on light and dark surfaces.

Do not scale font size with viewport width. Text must not overflow buttons, sidebar labels, status badges, or table cells.

Auth Identity Frontend

- /reset-password
- /verify-email
- /admin/login

Security UX:
- Never store tokens in localStorage.

Output:

- Loading, disabled, success, and error states.
- Tests for keyboard submission and error rendering.
04-B / Student Identity And Intake Consent
Design the student identity and consent flow.

Routes:
- /intake/profile
- /intake/education
- /intake/goals
- /intake/documents
- /intake/review

Steps:
1. Contact profile: name, email, phone, country.
2. Education background: current level, target program, grades summary.
3. Goals: destination countries, budget range, timeline, support type.
4. Documents status: passport, transcript, language test, visa records as status only unless upload policy exists.
5. Consent and review: explain data usage before submission.

Rules:
- Each step autosaves through backend endpoints.
- Sensitive fields explain why they are needed.
- Mask sensitive values after save.
- Do not send sensitive documents to AI APIs without explicit consent and policy.
- AI output is advisory metadata only.
04-C / Admin Roles And Access UI
Build the Team and Access page for admins.

- Invite teammate modal.
- Roles: Owner, Admin, Counselor, Finance, Viewer.

- Remove user confirmation dialog requiring typed organization name.

Frontend rules:

- Do not rely on frontend checks for authorization.
- Show audit log links for security-sensitive actions.
- Include empty states and skeletons.

Required tests:

- Backend 403 shows a clean error state.
- Invite form validates missing email and invalid role.

Prompt 05 - Booking And Checkout

05-A / Appointment Booking Flow
Build the student appointment booking flow.

Steps:
1. Select service.
2. Select counselor or "first available".
3. Pick date and time.
4. Fill intake summary.
5. Review service fee.
6. Create payment request.

Requirements:
- Mobile-first.
- Save draft after each step.
- Show timezone clearly.
- Validate required fields with zod.
- Use backend-created payment request; do not calculate final payment truth in the browser.
- Show privacy note before collecting PII.

Upgrade over AstroPay:
- Pair payment UX with education workflow context so students know what happens after paying.
05-B / Stellar USDC Hosted Checkout
Build hosted checkout at /pay/[publicId].

Layout:
- Mobile-first max width 480px.
- Ultraviolet logo, service summary, payment amount, and payment memo.
- "Secured by Ultraviolet" with a concise explanation: PII stays off-chain; payment is verified by backend.
- Freighter wallet action if available.
- Manual payment instructions with destination, asset, amount, and memo.
- Copy buttons for destination and memo.
- QR code if backend provides payment URI.

States:
- Loading request.
- Expired request.
- Waiting for payment.
- Submitted by wallet.
- Verifying on Stellar.
- Verified by backend.
- Failed verification.
- Backend unavailable.

Hard rules:
- The browser submits transaction evidence, then waits for backend verification.
- Do not show success just because a wallet submitted.
- Unknown transaction hash must not stay pending forever without retry or expiration copy.
05-C / Payment Result Screens
Create checkout result screens.

Success:
- "Payment verified" headline.
- Amount, asset, payment ID, and transaction hash.
- Receipt download if backend supports it.
- Next step: "Your counselor will review your intake."

Pending:
- Verification checklist: transaction exists, successful, destination, amount, asset, memo, hash uniqueness.
- Poll backend status every 3 seconds with backoff.

Failure:
- Human-readable reason such as wrong memo, wrong amount, wrong asset, duplicate hash, not found, or network unavailable.
- Try again.
- Contact support.
- Preserve checkout state.

Accessibility:
- Status changes announced via aria-live.
- No decorative success animation if reduce-motion is enabled.

Prompt 07 - Payment Operations

07-A / Payment Requests Manager
Build the Payment Requests page.

List:
- Search, filters, column visibility, pagination.
- Columns: Payment ID, Applicant, Service, Amount, Asset, Memo, Status, Expires, Transaction Hash, Created.

Create modal:
1. Select applicant and service.
2. Confirm amount and asset.
3. Generate backend-owned memo.
4. Review destination account and expiry.
5. Share payment link.

Validation:
- Backend generates payment ID and memo.
- Frontend cannot edit memo after creation.
- URL fields require HTTPS except localhost in development.

Statuses:
- created
- awaiting_payment
- submitted_for_verification
- verified_on_stellar
- failed_verification
- expired
- cancelled

Do not allow a verified payment to be downgraded from UI actions.
07-B / Verification Detail Drawer
Build the payment verification detail drawer.

Content:
- Expected destination.
- Expected amount.
- Expected asset.
- Expected memo.
- Submitted transaction hash.
- Backend verification result.
- Failure reason if any.
- Reconciliation attempts and timestamps.
- Duplicate hash warning if applicable.

Checklist display:
- Transaction exists on configured Stellar network.
- Transaction successful.
- Hash unused.
- Destination matches.
- Amount matches.
- Asset matches.
- Memo matches.
- Payment request ownership matches.
- Fee-bump, muxed account, path payment, and multi-operation policy handled.

Only backend data can check an item. The frontend must not infer checklist success locally.

Prompt 10 - Quality Gate

10-A / Frontend Acceptance Tests
Create frontend acceptance tests for ultraviolet-frontend.

Required checks:

- Student auth form validation.
- Admin login failure message does not leak account existence.
- Admin dashboard requires authenticated session.

Run:

If a command does not exist, add it or document why it is not available.
10-B / Design Review Checklist
Review the Ultraviolet frontend before shipping.

Reject the work if:
- It visually copies AstroPay too closely.
- It uses fake metrics without a demo label.
- It lets the frontend decide payment truth.
- It hides auth, identity, or admin access edge cases.
- It cannot distinguish local/testnet/production.
- Applicant tables break on mobile.
- Text overflows buttons or badges.
- It uses decorative blobs instead of useful product visuals.
- It lacks loading, empty, error, and unauthorized states.
- It introduces a contract feature without a backend-owned payment reason.
Brutal truth: a polished education-fintech frontend that lies about payment state is worse than a plain one. Trust is a system property, not a design style.
